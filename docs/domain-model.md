# Доменная модель и правила MVP

Этот документ закрывает этап 1 для MVP: основной продуктовый сценарий, состояния AI-решений и ключевые решения по модели данных, которые должны оставаться стабильными в ходе реализации.

## Основной пользовательский сценарий

1. Пользователь открывает приложение и создаёт новую заметку через отдельный editor/dialog.
2. Заметка сохраняется сразу после создания.
3. Каждая заметка всегда принадлежит ровно одной категории.
4. Новые заметки по умолчанию попадают в системную категорию `Без категории`.
5. После создания или редактирования AI асинхронно анализирует заметку относительно:
   - существующих категорий
   - существующих списков
6. AI возвращает один из трёх уровней уверенности:
   - `high`: применить автоматически
   - `medium`: показать предложение
   - `low`: ничего не делать
7. Любое автоматическое AI-действие должно быть обратимым через `Undo`.
8. Обычные destructive-действия в интерфейсе тоже имеют короткое окно `Undo`.

## Модель AI-решений

Это продуктовые состояния, а не обязательно enum’ы в базе данных.

- `high confidence`
  - AI достаточно уверен, чтобы автоматически назначить категорию или превратить заметку в пункт существующего списка.
  - В интерфейсе должен появляться toast с `Undo`.
- `medium confidence`
  - AI не должен действовать скрыто.
  - В интерфейсе должен появляться CTA с предложением действия.
- `low confidence`
  - Пользователь не должен получать лишних прерываний.
  - Не должно происходить автоматических изменений пользовательской структуры данных.

## Что AI может делать в MVP

- Назначать заметке одну категорию
- Превращать заметку в пункт существующего списка

## Что AI не должен делать в MVP

- Автоматически создавать списки
- Автоматически создавать категории
- Помещать одну заметку сразу в несколько категорий
- Менять порядок элементов списка эвристически

## Основные решения по данным

### User

- Владеет всеми заметками, категориями, списками и логами AI-действий

### Note

- Хранит исходный текст заметки
- Содержит необязательный `title` и обязательный `content`
- Всегда принадлежит одной категории
- Может быть в статусе `ACTIVE` или `ARCHIVED`
- Может опционально быть связана с одним или несколькими `list items`
- При сценарии `AI -> list item` исходная заметка архивируется, а не удаляется

### Category

- Только плоская структура
- Вложенности в MVP нет
- У каждого пользователя есть одна системная категория: `Без категории`
- Пользовательские категории уникальны по `(userId, name)`

### List

- Создаётся пользователем вручную
- Уникален по `(userId, name)`

### ListItem

- Принадлежит ровно одному списку
- Может опционально ссылаться на заметку, из которой был создан
- Ссылка на `noteId` используется в первую очередь для сценария, когда AI превращает заметку в пункт списка
- Дубликаты текстовых пунктов внутри одного списка запрещены через `(listId, text)`
- Повторное добавление одной и той же заметки в один список запрещено через `(listId, noteId)`
- Порядок элементов задаётся явно через `position`

### AiActionLog

- Хранит все применённые или потенциально откатываемые AI-результаты, важные для `Undo`
- Содержит:
  - тип действия
  - confidence
  - был ли результат применён автоматически
  - payload, необходимый для отката
- Low-confidence решения без действия можно не сохранять в MVP, если позже не потребуется отдельная отладка

## Важные продуктовые инварианты

- У заметки всегда есть ровно одна категория
- `Без категории` — это fallback-категория, а не отсутствие категории
- Списки являются ручными контейнерами; AI может только добавлять элементы в уже существующие списки
- Если AI превращает заметку в `list item`, исходная заметка не теряется, а переводится в `ARCHIVED`
- Автоматические AI-действия должны быть обратимыми
- Дубликаты элементов внутри одного списка запрещены

## Что было явно определено для MVP

- Ручное `Преобразовать заметку в список` остаётся отдельным CRUD/UI-поведением; при этом AI уже может превращать заметку в `list item` существующего списка
- Уровни confidence — это в первую очередь бизнес-правила; сами thresholds хранятся в backend-конфиге, а не в схеме БД
- Для простоты MVP используется одна таблица `AiActionLog`, без отдельной таблицы `UndoToken`
